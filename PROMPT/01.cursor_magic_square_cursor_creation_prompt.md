**Cursor**

마방진 구현 자체는 하지 않고, “커서”를 어떤 역할의 개념으로 잡으면 좋은지 설계 관점에서 설명드릴게요.


마방진에서 “커서”는 **현재 숫자를 넣을 위치를 기억하고, 다음 위치로 이동 규칙을 적용하는 역할**로 만들면 됩니다.

핵심 개념은 이렇게 잡으면 좋아요.

1. **커서가 기억할 값**
   - 현재 행 위치
   - 현재 열 위치
   - 이전 위치  
     충돌이 났을 때 되돌아가거나 아래로 이동하기 위해 필요합니다.

2. **초기 위치**
   - 홀수 크기 마방진이면 보통 첫 숫자 `1`은 **첫 번째 행의 가운데 열**에서 시작합니다.

3. **기본 이동 규칙**
   - 숫자를 하나 넣은 뒤 커서는 보통 **오른쪽 위 대각선**으로 이동합니다.
   - 즉, 행은 하나 위로, 열은 하나 오른쪽으로 이동하는 개념입니다.

4. **경계 처리**
   - 위로 나갔다면 맨 아래 행으로 이동합니다.
   - 오른쪽으로 나갔다면 맨 왼쪽 열로 이동합니다.
   - 이걸 “감싸기” 또는 “순환 이동”이라고 생각하면 됩니다.

5. **이미 숫자가 있는 칸을 만났을 때**
   - 오른쪽 위로 이동하려던 칸이 이미 채워져 있으면,
   - 방금 숫자를 넣었던 위치에서 **한 칸 아래로 이동**합니다.

6. **커서의 책임**
   - 숫자를 직접 넣는 역할보다는,
   - “다음 숫자는 어디에 들어가야 하는가?”를 계산하는 역할로 두는 게 깔끔합니다.

정리하면, 커서는 마방진 판 위를 돌아다니는 위치 관리자입니다.  
현재 위치를 알고 있고, “오른쪽 위로 이동”, “경계를 넘으면 반대편으로 이동”, “막히면 아래로 이동” 같은 규칙만 담당하게 만들면 됩니다.

`.cursorrules`의 빈 섹션을 MagicSquare 프로젝트용 규칙으로 채우고, 기존 `tdd_rules`도 요청한 하위 구조로 재작성하겠습니다. 먼저 현재 파일을 기준으로 전체 교체 범위를 확인하겠습니다.

이제 파일 전체를 하나의 일관된 YAML 구조로 정리하겠습니다. `forbidden`은 항목별 `pattern/reason/alternative` 구조로, `ai_behavior`는 실제로 따를 수 있는 체크 규칙으로 작성합니다.

```yaml
################################################################################
project:
  name: MagicSquare_01
  domain: MagicSquare
  language: Python
  goal: TDD와 ECB 패턴을 기반으로 유지보수 가능한 마방진 프로그램을 작성한다.

################################################################################
code_style:
  python_version: "3.10+"
  style_guide: PEP8 엄격 준수
  type_hints: 모든 함수 파라미터와 반환값에 필수
  docstring: Google 스타일, 모든 public 메서드에 필수
  max_line_length: 88

################################################################################
architecture:
  pattern: ECB
  layers:
    boundary:
      responsibility: 외부 입출력 담당
      examples:
        - UI
        - API
        - CLI
    control:
      responsibility: 비즈니스 로직 담당
      examples:
        - 마방진 생성 흐름 제어
        - 입력 검증 조율
        - entity 규칙 호출
    entity:
      responsibility: 도메인 데이터 및 규칙 담당
      examples:
        - 마방진 보드
        - 좌표
        - 마방진 생성 규칙
  dependency_direction:
    allowed:
      - boundary -> control
      - control -> entity
    forbidden:
      - entity -> control
      - entity -> boundary
      - control -> boundary

################################################################################
tdd_rules:
  red_phase:
    description: 구현 전에 실패하는 테스트를 먼저 작성하는 단계
    rules:
      - 요구사항을 검증하는 테스트를 먼저 작성한다.
      - 테스트가 실제로 실패하는지 확인한 뒤 다음 단계로 진행한다.
      - 실패 원인이 테스트 대상 기능의 미구현 때문인지 확인한다.
    must_not:
      - 테스트 없이 프로덕션 코드를 먼저 작성하지 않는다.
      - 실패를 확인하지 않고 green_phase로 넘어가지 않는다.
      - 여러 요구사항을 한 번에 테스트하지 않는다.
  green_phase:
    description: 실패한 테스트를 통과시키기 위한 최소 구현 단계
    rules:
      - 현재 실패한 테스트를 통과시키는 최소 코드만 작성한다.
      - 새 기능 범위를 테스트 요구사항 밖으로 확장하지 않는다.
      - 모든 테스트가 통과하는지 확인한다.
    must_not:
      - 이 단계에서 리팩터링하지 않는다.
      - 불필요한 추상화나 일반화를 추가하지 않는다.
      - 테스트와 무관한 코드를 수정하지 않는다.
  refactor_phase:
    description: 기능 변경 없이 코드 구조를 개선하는 단계
    rules:
      - 외부 동작을 바꾸지 않고 중복과 복잡도를 줄인다.
      - ECB 레이어 경계를 유지한다.
      - 기존 테스트 커버리지를 유지한다.
    must_not:
      - 새로운 기능을 추가하지 않는다.
      - 테스트 커버리지를 낮추지 않는다.
      - 통과 중인 테스트를 깨뜨린 상태로 종료하지 않는다.

################################################################################
testing:
  framework: pytest
  pattern: AAA (Arrange-Act-Assert)
  coverage_minimum: 80%
  fixture_scope:
    function: 테스트 간 상태 공유를 막기 위한 기본 scope
    module: 생성 비용이 크고 불변인 테스트 데이터에만 허용
    session: 외부 자원 초기화처럼 전체 테스트에서 공유해도 안전한 경우에만 허용
  naming_convention: test_ 접두사 필수

################################################################################
forbidden:
  - pattern: print()
    reason: 테스트와 로그 제어가 어렵고 boundary 책임을 흐릴 수 있다.
    alternative: boundary 레이어에서 명시적인 출력 함수 또는 logging을 사용한다.
  - pattern: 하드코딩 상수
    reason: 마방진 크기, 이동 방향, 검증 기준이 코드 곳곳에 흩어져 변경이 어려워진다.
    alternative: 의미 있는 이름의 상수나 설정값으로 분리한다.
  - pattern: except 단독 사용
    reason: 예외 원인을 숨겨 디버깅과 테스트 실패 분석을 어렵게 만든다.
    alternative: 구체적인 예외 타입을 명시하고 필요한 경우 메시지나 로깅을 남긴다.

################################################################################
file_structure:
  root: MagicSquare_01
  # MagicSquare_01/
  # |-- boundary/
  # |   |-- cli.py
  # |   `-- __init__.py
  # |-- control/
  # |   |-- magic_square_controller.py
  # |   `-- __init__.py
  # |-- entity/
  # |   |-- magic_square.py
  # |   |-- position.py
  # |   `-- __init__.py
  # `-- tests/
  #     |-- boundary/
  #     |-- control/
  #     `-- entity/
  directories:
    boundary: 외부 입출력과 사용자 인터페이스 코드를 둔다.
    control: 유스케이스 흐름과 비즈니스 로직 조율 코드를 둔다.
    entity: 마방진 도메인 데이터와 규칙 코드를 둔다.
    tests: 프로덕션 코드 구조를 반영한 pytest 테스트를 둔다.

################################################################################
ai_behavior:
  before_coding:
    - 코드 작성 전 관련 테스트 파일을 먼저 확인한다.
    - 변경할 기능이 어느 ECB 레이어에 속하는지 판단한다.
    - 기존 public 메서드의 타입힌트와 docstring 규칙을 확인한다.
  during_coding:
    - ECB 레이어 경계를 위반하지 않는다.
    - 타입힌트 없는 함수를 생성하지 않는다.
    - 테스트 요구사항을 벗어난 기능을 추가하지 않는다.
    - tdd_rules 위반이 감지되면 사용자에게 경고한다.
  after_coding:
    - 관련 pytest 테스트 실행을 우선 제안하거나 수행한다.
    - 테스트 실패가 있으면 원인을 요약한다.
    - 리팩터링 후 기능 변경이 없었는지 확인한다.
```

`.cursorrules`를 기준으로 ECB 구조와 TDD 규칙을 확인한 뒤, 먼저 관련 테스트 파일을 만들고 그 다음 `User` 엔티티를 최소 구현하겠습니다. 기존 프로젝트 구조가 있는지도 함께 확인하겠습니다.

현재 프로젝트에는 `.cursorrules`만 있어서 ECB 폴더와 테스트 폴더를 새로 만들어야 합니다. TDD 순서에 맞춰 테스트 파일을 먼저 작성하고 실패를 확인한 뒤 엔티티를 구현하겠습니다.

테스트는 `tests/entity/test_user.py`에 먼저 작성하겠습니다. `User`는 엔티티 레이어의 도메인 객체로 두고, 이름 정규화와 빈 이름 거부 같은 최소 도메인 규칙부터 검증하겠습니다.

`red_phase`에 맞춰 테스트 파일만 먼저 만들었습니다. 이제 pytest를 실행해서 구현 전 실패 상태를 확인하겠습니다.

**Cursor**

현재 Python 기준으로 작성된 `.cursorrules`를 읽고, Java 프로젝트에서 Cursor가 사용할 수 있도록 규칙을 주제별로 분리하겠습니다. 기존 내용을 기준으로 Java 코드 스타일, ECB 아키텍처, TDD/테스트, 금지 패턴, AI 행동 규칙으로 나누겠습니다.

`.cursorrules`는 단일 YAML 파일이고, Cursor의 최신 프로젝트 규칙은 `.cursor/rules/*.mdc`로 나눠 쓰는 형태가 더 적합합니다. 기존 Python 규칙을 Java 기준으로 바꿔 주제별 `.mdc` 규칙 파일로 분리하겠습니다.

이제 Java 기준 규칙을 `.cursor/rules/` 아래에 섹션별로 나눠 만들겠습니다. Python 전용 항목은 Java 17+, Google Java Style, JUnit 5, Javadoc, 명시적 타입 규칙으로 치환합니다.

규칙 파일 분리는 완료했습니다. 마지막으로 생성된 `.mdc` 파일들에 진단 오류가 있는지 확인하겠습니다.

Python 기반 `.cursorrules`를 읽고 Java 기준 Cursor Rules로 분리했습니다.

생성한 파일:
- `.cursor/rules/project.mdc`
- `.cursor/rules/java-code-style.mdc`
- `.cursor/rules/ecb-architecture.mdc`
- `.cursor/rules/tdd-rules.mdc`
- `.cursor/rules/testing.mdc`
- `.cursor/rules/forbidden-patterns.mdc`
- `.cursor/rules/ai-behavior.mdc`

주요 변경 기준:
- Python 3.10+/PEP8/pytest 규칙을 Java 17+/Google Java Style/JUnit 5 기준으로 변환
- ECB 구조를 Java 패키지 구조 기준으로 분리
- TDD, 금지 패턴, AI 행동 규칙을 각각 독립 `.mdc` 파일로 분리
